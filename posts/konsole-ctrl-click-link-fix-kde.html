Fixing Ctrl+Click Links in KDE Konsole (kioexec Cache Bug)

Ctrl+clicking a URL in Konsole opens file:///home/you/.cache/kioexec/krun/.../unnamed instead of your browser? Here's why and how to fix it.

Symptoms Any Ctrl+click on a URL in Konsole (or any KDE app launching a browser via xdg-open) results in a cached HTML file being opened from ~/.cache/kioexec/ instead of loading the actual URL in your browser. The browser opens with a path like file:///home/user/.cache/kioexec/krun/26256_0/unnamed.

The Root Cause

KDE's kioexec is a KIO worker that intercepts xdg-open calls for HTTP/HTTPS URLs. Instead of passing the URL directly to your browser, it downloads the page to a temp file under ~/.cache/kioexec/ and opens that. This is meant for FTP or local file handling - but it misfires on regular web links when the BROWSER environment variable is unset or points to a binary that doesn't exist.

The Fix (3 Methods)

Method 1: Set the BROWSER Env Var

# Standard installs:
export BROWSER="firefox"

# Flatpak:
export BROWSER="flatpak run org.mozilla.firefox"

# Custom path (e.g. Zen Browser in /opt):
export BROWSER="/opt/zen/zen"
echo 'export BROWSER="/opt/zen/zen"' >> ~/.bashrc

Method 2: Create a .desktop File

cat > ~/.local/share/applications/my-browser.desktop << 'EOF'
[Desktop Entry]
Name=My Browser
Exec=/opt/zen/zen %u
Terminal=false
Type=Application
MimeType=x-scheme-handler/http;x-scheme-handler/https;text/html;
EOF

xdg-settings set default-web-browser my-browser.desktop
xdg-mime default my-browser.desktop x-scheme-handler/http
xdg-mime default my-browser.desktop x-scheme-handler/https

Method 3: Disable Ctrl+Click in Konsole

Konsole → Settings → Configure Konsole → General
→ Uncheck "Enable Ctrl+Click for URLs"

Why This Happens

KDE Plasma uses kioexec as a generic protocol handler. When you Ctrl+click a URL in Konsole, Konsole asks the system to open the URL via xdg-open. On KDE, xdg-open delegates to kioclient exec, which hands it to kioexec. kioexec sees an HTTP URL, thinks "I should download this file first," fetches it into ~/.cache/kioexec/, and tells the system to open the cached copy.

The result: instead of your browser loading https://example.com/login, it loads file:///home/you/.cache/kioexec/krun/12345_0/unnamed - which is a stale or broken HTML snapshot.

Why method 1 works
Many CLI tools (dv login, gh auth login, etc.) check the $BROWSER environment variable first. If it's set to a valid browser binary, they call it directly instead of going through xdg-open, bypassing kioexec entirely.

For CLI Auth Flows (Alternative)

If a tool's browser-based login still fails even after fixing this, skip the browser redirect entirely by using an API token: 

# Example for Diversion (dv)
# 1. Log in via the web UI manually
# 2. Go to Settings → API Tokens → Generate
# 3. Use the token directly:
dv authenticate <your-token>
